# ADR-0001：积分兑换改为账号级统一钱包

- 状态：已采纳（2026-06-08）
- 取代：PRD v1 §5.3 / §8 中「积分与报销额度按学科隔离、不合并兑换」的原决定

## 背景

平台多学科并行，原设计把**积分与报销额度按学科隔离**：每个学科一个钱包，兑换绑定 `subjectId`，余额按学科求和。实践中这造成「学科墙」——员工在 A 课攒 40 分、B 课攒 30 分，买不了 60 分的书，体验割裂；兑换页还得带学科 tab 切来切去。

而「赚分」按学科隔离是有意义的（进度、章节排名、基础积分都应按课独立核算）；只有「花钱（兑换）」这步没必要分科。

## 决定

**赚分按学科隔离，钱包/兑换账号级统一。**

- **赚分侧**（`BASE` / `RANK_BONUS` 流水）继续记 `subjectId` 作为**来源**——「每门课发了多少分」仍可统计；进度、章节排名不变。
- **花费侧**：`Redemption` 与 `REDEEM` 流水改为**账号级**，`subjectId` 置空（schema 中两者 `subjectId` 改为可空）。
- **余额** = 该用户**全部流水之和（跨学科）**：`getAccountBalance(userId)`；可用余额 = 账号余额 − 待审批占用。
- 兑换页去掉学科 tab，一个余额、一次申请。财务统计（`getFinanceStats`）改为**全公司**口径。

## 影响

- 员工可合并各门课的积分自由兑换；兑换页更简单。
- 来源信息不丢：赚分流水仍带 `subjectId`，可回答「某课产生了多少积分预算」。
- 管理后台「员工学情」花名册的 `积分` 列在 per-subject 口径下自然变为「**本学科赚得**」（REDEEM 无 subjectId，不参与 per-subject 求和）。
- **前提**：适用于各课报销预算来自**同一个池子**。若未来出现「不同课程预算来自不同口袋、财务上必须分开」，需另立 ADR 回到（或部分回到）按学科核算。
- 历史数据：旧 `Redemption`/`REDEEM` 行的 `subjectId` 保留不动（账号余额按全部流水求和，旧值不影响结果）。
