# AI 学习平台 PRD（多学科可扩展）

> 基于麦肯锡「了解一个行业」方法论的游戏化内部学习系统。
> 文档状态：v1 共识稿（grill-me 访谈结论），待开发。
> 最后更新：2026-06-04

---

## 1. 产品概述

### 1.1 一句话定位
一个面向内部员工的游戏化学习平台：系统内置某学科的 100 个关键词（分 5 章），员工每周完成一章，针对每个关键词提交外部学习笔记，由 DeepSeek 进行 1–100 分打分与追问，达标即得积分，积分可按 1:1 兑换报销额度；每章前 3 名额外奖励。系统设计为多学科可扩展（AI / 医学 / 心理学等），可由管理员同时上线多个学习主题，员工自由选择学习哪个（各学科进度与积分按学科隔离）。

### 1.2 方法论来源
借助麦肯锡了解一个行业的关键步骤之一——「总结行业的 100 个关键词」——被产品化为：系统给词、员工自行检索总结、AI 评估并追问、积分激励。

### 1.3 目标与非目标
**目标**
- 跑通从「登录 → 看关键词 → 提交笔记 → AI 打分/追问 → 得积分 → 兑换」的完整真实闭环。
- 用游戏化体验降低学习枯燥感，用排名与积分驱动内部比拼与互相学习。
- 数据模型与学科解耦，未来加新学科只需录入内容，无需改代码。

**非目标（v1 不做）**
- 不做笔记是否为 AI 代写的检测。
- 不做复杂的财务对账系统（兑换审批后仅记录，不对接报销系统）。
- 不做移动端原生 App（用响应式 Web 覆盖移动端）。

## 2. 技术与部署

| 项 | 决策 |
|---|---|
| 形态 | 真功能薄切片（非静态稿），接真 DeepSeek |
| 框架 | Next.js（App Router）全栈 |
| 数据库 | Postgres + Prisma ORM |
| AI 评分 | DeepSeek API，生产用真实 key（放 `.env`，不入库）；测试与本地演示可切换 Mock 评分器 |
| 规模 | 中等公司 50–500 人，云部署 |
| 终端 | **PC 与移动端均完整兼容**（响应式，移动端为一等公民，非仅可视） |
| 语言 | 全站中文 |

**AI 评分服务抽象**：定义统一的 `ScoringService` 接口，提供两个实现：
- `DeepSeekScoringService`（生产）
- `MockScoringService`（自动化测试 + 本地演示，确定性打分 + 模拟追问）
通过环境变量 / 配置切换，业务代码不感知具体实现。

## 3. 角色与权限

| 角色 | 能力 |
|---|---|
| 员工 | 学习、提交笔记、答追问、查看自己进度/积分、查看公开排行榜与（完成后）他人笔记、发起兑换申请、修改密码 |
| 管理员 | 员工的全部能力 + 名单管理、学科/章节/关键词录入、上线/下线学科（可多学科并行）与各自设开始日、审批兑换、查看全部统计、（演示）手动结算/快进一周 |

- 角色用用户记录上的 `role` 字段区分（`employee` / `admin` / `superadmin`）。
- 系统初始化时预置 1 个超管账号（固定账号 + 初始密码）。

## 4. 身份与登录

### 4.1 注册激活流程
1. 管理员预导入员工名单（白名单，含姓名、可选手机号等）；**全员共享默认密码为 `Aa123456!`**（管理员可重置）。
2. 员工首次用「姓名或已登记手机号 + 共享默认密码」登录 → 自动激活账号。
3. 首次登录**强制修改密码**后方可进入。
4. 名单之外的姓名/手机号无法登录；当前不开放自助注册，但手机号字段需为后续注册登录预留。

### 4.2 边界与规则
- 重名处理：名单导入时若姓名重复需提示管理员消歧（如加工号 / 部门后缀），保证登录姓名唯一。
- 手机号：可选填，填则全局唯一；管理员录入/导入后，员工可用手机号代替姓名登录。
- 密码：首登强制改密；员工可随时在「我的」里改密。
- 默认密码可由管理员重置（针对忘记密码或未激活员工）。

## 5. 内容模型（学科可扩展）

### 5.1 层级
```
Subject（学科） 1 ── N Chapter（章节，固定 5 章）
Chapter        1 ── N Keyword（关键词，每章 20 个，全学科共 100 个）
```

- 每个学科：固定 5 章、每章 20 词、共 100 词、对应 100 基础积分体系。
- 章节有主题，整体编排「从起源到最新发展趋势」的脉络。
- 关键词字段：`term`（词）、`description`（可选简介）、`referencePoints`（**可选**「参考考核要点」，辅助 AI 打分；不填则 AI 自由判断）。

### 5.2 内容录入
- **预置数据**：开发时由我预先生成一套真实的「AI 行业 100 关键词 + 5 章主题」入库作为种子数据。
- **录入 UI**：管理员可新建学科、编辑章节主题、增删改关键词及其参考要点（为未来医学 / 心理学等学科准备）。

### 5.3 学科上线（多学科并行）
- 平台可**同时上线多个学习主题**（如「人工智能（普及版）」「人工智能（专业版）」）。管理员对每个学科独立「上线/下线」并设定其**开始日**；员工在学习页自由选择学习哪个学科。
- 学科对员工可见 = 已上线（`isActive`）且未归档；每个学科有**各自独立的开始日与自然周解锁节奏**。
- 每个学科的**进度、章节排名、基础积分赚取按学科隔离**独立核算；但**积分钱包与兑换为账号级统一**——各门课赚的分汇入同一余额、可合并兑换（赚分流水仍保留来源学科）。见 §8.2 与 `docs/adr/0001-unified-redemption-wallet.md`。

## 6. 核心学习闭环

针对每个关键词：
1. 系统展示关键词（及可选简介）。
2. 员工在外部检索、总结后，在输入框填写**学习笔记**（纯文本；**最少 100 字、最多 2000 字**，超出范围则禁止提交并提示）。
3. 提交 → **流式等待**：实时显示 AI 的**思考过程**（reasoning），完成后进入下一步。最终的 JSON 结果只在后台解析落库，**不展示给用户**。
4. DeepSeek 返回：
   - **初始分**（1–100）——提交后即在答追问页**显著显示**。
   - 根据笔记薄弱点**动态生成 1–3 个追问**（笔记越好问得越少）。
5. 员工逐一回答追问（**每条回答 ≤ 1000 字**）；答追问时仍可对照自己刚写的笔记。
6. DeepSeek 综合「原笔记 + 追问回答」给出**最终分**。
7. 判定：
   - **最终分 ≥ 60** → 该关键词完成，记 **1 积分**。
   - **最终分 < 60** → 未通过，可**无限次重新提交笔记**重走打分流程，**取历史最高分**为准。
8. API 调用失败 → 友好错误提示 + 重试按钮（不扣次数、不丢笔记草稿）。

### 6.1 打分依据（评分 rubric）
- 统一的通用评分维度，例如：准确性、深度、完整性、条理性、原创思考等（具体权重在实现时固化为 Prompt）。
- 若该关键词填了「参考考核要点」，则作为额外上下文提升打分准确度。

### 6.2 重提与提分
- 已通过（≥60）的关键词仍可重新提交以刷新更高分（用于提升章节平均分、冲排名），同样取历史最高分。

### 6.3 每日上限
- **每天最多新完成 10 个关键词**（按本地自然日的 `KeywordProgress.completedAt` 计数）。达上限后**当天不能再开新词**（写笔记页直接提示「今天已完成 10 个，明天再来」），服务端 `startAttempt` 同样拦截。
- **重刷已通过的词不受限**（不算新完成）；已开始（待答）的词也能答完。

## 7. 节奏与排名

### 7.1 周期机制
- 管理员开启学科时设**开始日**；以**自然周（周一~周日）**为边界。
- **顺序解锁**：第 1 周解锁第 1 章，第 2 周解锁第 2 章……不能跳章。
- 本周内可**提前**把当周章节干完，但不能提前进入下周章节。

### 7.2 周结算（每周日夜）
- 结算上周章节排名：**须完成该章全部 20 个关键词且做完该章「章节反思」**（截至本周末）才进入排名。
- 排序依据：该章 20 个关键词**最终分的平均值**，取 **top3**，各额外奖励 **+100 积分**。
- **并列处理**：并列名次**均给 +100 奖励**（不因并列稀释、不限人数）。
- 注意：**反思只是「各章冠军 / 周结算」的门槛**；**学习榜（按词均分的总榜）不要求反思**，通关一个词即上榜（见 §9）。
- 完成整章第 20 个关键词后，关键词**结果页给出醒目提示**引导去做章节反思（说明「做完才参与本章周冠军结算」）。

### 7.3 补进与错过（含中途加入 / 周期结束后的补学）
- 已解锁的旧章**保持解锁**，员工可随时回去补完剩余关键词、**正常赚取基础积分**（每词 ≥60 得 1 分）。
- 但 top3 排名是**当周截止**的（结算只统计「该章自然周周末前完成且已反思」者，见 §7.4），错过当周不补排名、不补奖励。
- **中途加入的新员工**：能学全部已解锁章节、照常拿基础积分；对**已过去的章节**自然不参与排名（时间窗已截止），无需单独标记身份。
- **周期结束**：当统一进度走过最后一章（当前周 > 章节数），该学科的「学习/培训/奖励周期」即结束。此后学科**保持上线供补学**——补学仍记基础积分、可兑换，但不再有新的章节排名。学员侧显示「培训周期已结束 · 补学不限时（不参与排名）」。注意：**「结束」≠「下线」**；下线（isActive=false）会让学科从学员列表消失、谁都不能再学。

### 7.4 结算触发
- **生产**：定时任务（cron / scheduled job）在周日夜自动解锁下章 + 结算上章排名。
- **原型 / 演示**：额外提供「手动结算本章 / 快进一周」按钮，便于演示与测试。

### 7.5 结业证书
- 某学科 **100 个关键词全部通过**后，学习地图出现「你已结业」入口，进入**结业证书**页 `/learn/:subjectId/certificate`。
- 证书含：姓名、学科、通过数（100/100）、平均分、完成日期、章节数与章节反思完成数、确定性证书编号；5 章反思全部完成则额外标注「含章节反思」。
- 提供「打印 / 保存为 PDF」（走浏览器原生打印，打印时隐藏导航与按钮）。未学完者访问证书页将被挡回该学科地图（不泄漏证书）。

## 8. 积分与兑换

### 8.1 积分规则
- **基础积分**：每个关键词完成（≥60）记 1 分，单学科最多 100 分。
- **排名奖励**：每章 top3 各 +100 分，最多 5 章 → 最多 +500 分。
- 单学科单人理论上限 ≈ **600 积分**。

### 8.2 兑换规则
- **1 积分 = 1 元**报销额度。
- **可多次部分兑换**（如先用 50 元买书，余额继续累积）。
- **统一钱包**：各门课赚的积分进**同一个账号余额**，兑换不绑学科、可合并使用（赚分仍按学科隔离，REDEEM 流水为账号级、不挂学科）。详见 `docs/adr/0001-unified-redemption-wallet.md`。

### 8.3 兑换流程
1. 员工发起兑换申请：填写**物品 / 工具名称 + 类别（书籍/工具/课程·会员/其他）+ 金额**，并可**上传报销凭证**（截图 PNG/JPG/WebP 或 PDF，≤5MB）。凭证存库、经鉴权接口下载，仅**本人或管理员**可查看。
2. 管理员**审批**（通过 / 驳回），审批时可查看上传的凭证。
3. 通过后**扣减积分**并记录流水；驳回则不扣分。
- 余额、申请状态、历史流水在员工端与管理端均可查。

### 8.4 共享兑换目录（「大家兑换了什么」）
- 兑换页内嵌一块**全员已通过（APPROVED）兑换的共享目录**，**跨学科**（不随学科 tab 变）：展示**物品名 · 类别 · 持有人 · 金额 · 日期**，可按**类别筛选 + 关键词搜索**（物品/持有人）。买了书或工具的人公开可见，方便同事**借阅 / 共用**（联系走线下，平台不做借还流转）。凭证仍仅本人/管理员可见，目录只公开物品层面的信息。
- **反馈**：任何员工都可对某条目留一条反馈（短文本 ≤500 字 + 可选**推荐 / 一般 / 不推荐**倾向），持有人本人也可；反馈展示在该条目下，供大家决定是否值得借/用。落库 `RedemptionFeedback`（挂在该条兑换下）。

## 9. 社交与统计

### 9.1 防抄袭的同伴可见性
- **关键规则**：只有员工**自己完成（≥60）某关键词后**，才能查看其他人该关键词的笔记。
- 完成前看不到任何人该词的答案；完成后解锁同词他人记录（笔记 + 追问与回答），**只展示得分最高的 3 位**，供学习参考。
- **排行榜可点进 top10 成员的「闯关记录」**（`/leaderboard/[userId]`）供观摩：守同款防抄袭——每条笔记/回答仅当**观看者本人也完成了该词**才解锁，否则只给词名+分数并提示「完成本词后可见」（锁住的内容服务端不下发）。仅在榜 top10 可看（与 §9.2 一致）。
- **该页同时展示对方的「当前画像」（仅正向公开，不含成长时间线）**：强项 + 兴趣 + 画像全文（服务端用 `stripSensitivePortrait` **剔除「待加强 / 盲区」小节**再下发）。即看得到他人擅长什么、在学什么，但**不公开其短板与盲区**（个人敏感），也不展开逐次更新轨迹（成长轨迹仅本人可见）。对方还没有画像则不显示该区。

### 9.2 公开身份策略
- 排行榜 / 公开统计中：**只公开靠前者 / 完成者的真名 + 头像**（荣誉感）。
- **不向全员公开落后者与低分**；落后提醒仅管理员可见。

### 9.3 管理后台统计（四类）
1. **全员进度总览**：完成率、已开始 / 落后等**汇总数字**（逐人花名册、落后者明细与逐人钻取见 §9.3.1「员工学情」，数据统计页只保留大盘聚合，不再逐人罗列）。
2. **每章排行榜 / top3 获奖名单**：各章平均分排名与奖励发放情况。
3. **分数 / 质量分析**：平均分、普遍低分的「难词」、分数分布、追问难度等。
4. **积分与兑换财务统计**：总发放积分、已兑换金额、待审批申请、预算消耗。

### 9.3.1 全员学情（逐人 360°）
- 后台「**员工学情**」页（`/admin/learners`）：可**搜索/筛选/排序**的全员花名册（按学科切换），逐人一行展示**部门、进度、均分、积分、状态、最近活跃**；快捷筛选「全部 / 落后者 / 未激活」。落后者 = 已激活但完成数低于「当前周之前各章本应学完」的线。
- 点进某员工 → 逐人 **360° 详情**（`/admin/learners/[userId]`）：基本资料、**跨学科进度**分项、**完整画像**（管理视图**不脱敏**，含待加强/盲区）、**成长轨迹**时间线（画像按行 git-diff，与 §14.5 同款渲染）、以及**逐词答题记录**（笔记 / 追问 / 分数 / AI 反馈）。
- 与 §9.2 同伴可见性是**两条独立路径**：同伴互看脱敏 + 词锁（`social.ts`），管理视图看全（§9.3 管理员可看全部统计），互不复用、互不削弱。

### 9.4 员工端统计
- 员工在**完成关键词 / 章节后**，可查看对应的部分统计与（解锁的）他人答题详情，形成互相学习与比拼氛围。
- **学习榜排名按「每个词最高分的平均分」**（保留两位小数）降序，而非积分——积分人人都是「每词 1 分」差不多，用均分体现质量；并列按完成词数。通关一个词即上榜。积分仍用于兑换，不参与排名。

## 10. UI / 视觉

- **风格**：游戏化闯关风。
- **元素**：5 章 / 100 词做成闯关地图、进度条、勋章、积分获取动画、排行榜、明亮活泼配色。

- 提交等待、得分揭晓、积分到账、升章解锁等关键节点配合动效，强化正反馈。

## 11. 交付物与测试

### 11.1 演示数据
- 预填：样例管理员 / 超管、若干样例员工、AI 学科 100 词、部分已答题与积分数据。
- 提供「从登录到兑换」可点击走通的完整演示路径。
- 提供演示用「手动结算 / 快进一周」入口。

### 11.2 自动化测试
- 对核心逻辑写自动化测试（使用 Mock 评分器，不真打 API）：
  - 打分判定（≥60 通过 / <60 重提 / 取最高分）
  - 积分累计（基础 + 排名奖励）
  - 章节解锁与顺序约束

## 14. 个性化追问与员工画像（v1.1 增补）

把「照本宣科」变成「结合岗位、逐步深入」的个性化引导。

### 14.1 员工资料（onboarding）
- 员工首登改密后，先填一次基本资料（6 项）：岗位、部门、职级/年限、专业背景、对 AI 的熟悉度、最想把 AI 用在哪些工作。
- 未填资料的员工进入 `/learn` 会被引导到 `/onboarding`；管理员不需要。

### 14.2 员工记忆（标签 + 画像）
- 每个员工有一份系统维护的记忆：**结构化标签**（强项 / 薄弱点 / 兴趣方向 / 知识盲区）+ 一段**持续重写的画像摘要**（Markdown 文档，固定小节结构便于按行 diff）。
- **更新时机**：每个关键词**终评后增量更新**（在已有标签基础上演进，不抹掉历史）。
- **权限**：个人**资料可由员工编辑**；**AI 画像只读**（员工只能看，不能改）。

### 14.3 个性化追问
- **不做刻意引导**：把「员工资料 + 记忆」拼进追问/评分提示，交给大模型自然个性化——随学习推进，追问越来越贴合其角色与定位。
- 打分服务契约（`ScoringService`）相应扩展：`submitNote` / `finalize` 接收可选 `LearnerContext`（资料+记忆）；新增 `updateMemory(本次表现 + 当前记忆) → 新标签+画像`。Mock 实现保持完全确定性。

### 14.3.1 追问提示原则（修正）
- 单个关键词的追问提示**只把档案+记忆作为上下文拼进去**，交给模型自由发挥；**不写「据背景调深浅/不强行联系岗位」这类元指令**（会束缚模型）。岗位结合题只在章节总结环节出现（§14.4）。

### 14.4 章节总结与反思（每章学完后）
- 完成某章全部关键词后，增加一个「**章节总结 + 追问**」环节；此时显式加入「如何把本章所学**结合到你的岗位 / 实际工作场景**」类问题，帮助员工把知识落到工作中。
- **与单词提交环节同款体验（统一风格）**：提交反思作答 → AI 生成本章小结**走流式**（Node 路由 `/api/learn/reflect`），等待时实时显示 DeepSeek 的**思考过程**（同款 `ThinkingPanel`）；JSON 结果（summary/portrait）只后台解析落库不外泄，完成后 `router.refresh()` 进小结页。小结的 reasoning **落库**（`ChapterReflection.reasoning`），可在小结旁的「**AI 思考过程**」小弹窗回看（与初评/终评同款 `ReasoningDialog`）。小结按 Markdown 渲染。
- **健壮性**：出反思题（`reflectionQuestions`）若遇模型偶发非法 JSON / 失败，**兜底为结合岗位的通用反思题**，不再让整页崩（失败仍进 AI 审计日志）。

### 14.5 画像快照与成长轨迹
- 每次画像更新都存一份**快照**（EmployeeMemorySnapshot：触发的关键词、最终分、当时的标签与画像、相对上一版的 diff、序号 seq）。
- 单独的**成长轨迹页**（`/growth`）以**线性卡片时间线**展示画像如何随学习一步步变清晰；每张卡片显示触发关键词+得分、新增标签 chips、以及画像的 **git 风格按行 diff**（画像是 Markdown，逐行对比最直观）。`/profile` 页可编辑资料 + 只读查看当前画像。

## 15. 体验增补（v1.2）

v1.1 之后落地的增补，均不改变核心闭环与积分规则，只增强可审计性与体验。

### 15.1 AI 调用审计（管理员可见）
- 每次 AI 调用（提交笔记初评 / 终评 / 画像更新 / 章节反思出题与总结）都落一条审计日志：完整 system / user 提示（含拼入的笔记、追问回答、学习者档案、参考考核要点、章节主题等上下文）、模型**推理过程**（reasoning）、返回、token 用量、耗时、成功或失败原因。
- 管理后台 `/admin/ai-logs` 按阶段筛选、逐条展开查看；日志独立留存（`AiCallLog`，不设外键级联），不随用户/关键词删除而消失。
- 页首「**用量统计**」（全部记录累计）：总调用数、token 总量（输入 / 输出 / 推理分项）、失败数；**按调用类型**（各阶段调用次数 + token，带占比条）与**按日**（近 14 天，`date_trunc` 聚合）两个明细。全程展示「token」全称（不用「tok」缩写），数字千分位。
- 实现：编排层用 `AsyncLocalStorage` 注入调用归属（谁 / 哪个词或章 / 哪个阶段），打分服务底层（DeepSeek `chat()`）统一记录；写日志为 best-effort，失败不影响评分。Mock 不产生真实提示，不记录。

### 15.2 中途退出保留记录
- 写笔记过程中**防抖自动保存草稿**（服务端 `NoteDraft`，跨设备 / 跨刷新）；重新进入预填并提示「已恢复草稿」，成功提交后清除。
- 答追问过程中**回答同样自动保存**（落到既有 `Followup.answer`），中途退出再回来不丢；终评时以最终回答覆盖。

### 15.3 员工完整答题记录
- 终评结果页展示「**本次答题记录**」：笔记原文 + 每条追问与回答 + 初评 / 终评 + AI 反馈。
- 多次重提时提供「**我的答题记录（全部 N 次）**」可展开归档，逐次回看。

### 15.4 追问的内容依据（澄清）
- 单词追问由模型对照「**参考考核要点 + 章节主题 + 本篇笔记**」找出缺失 / 含糊 / 可深入处生成，**扣住笔记本身、非随机**；仍遵守 §14.3.1（只给上下文、不写「据背景调深浅」类元指令）。岗位结合题只在章节总结环节（§14.4）出现。

### 15.5 管理员也可参与学习 + 顶栏导航梳理
- 管理员 / 超管不再被挡在学习侧之外：可进入 `/learn` 闯关答题，并能填资料、看成长、兑换（无资料先引导 `/onboarding`）。打分上下文在管理员无资料时自然降级为通用，不报错。
- 顶栏导航按角色梳理、两类结构一致，便于切换：
  - 员工：闯关 / 排行榜 / 兑换 / 成长 / 我的。
  - 管理员：管理后台 / 闯关 / 排行榜 / 兑换 / 成长 / 我的。「管理后台」为后台首页，汇总名单 / 内容 / 排名 / 兑换审批 / 统计 / AI 记录（卡片入口），不再把后台子页平铺到顶栏。
- 同伴笔记演示数据（`seed`）改用关键词**真实简介 + 考核要点 + 岗位口吻**生成，替换原「…分水平的学习笔记示例」占位模板；线上同伴笔记本就取真实提交，此处仅修正演示数据观感。

### 15.6 结果页向 AI 追问（v1.3）
- 答完追问、进入结果页后，员工可在「**向 AI 追问**」处就该词继续提问（每条 ≤ 1000 字）；AI 结合该次**笔记 + 追问回答 + 此前提问**（多轮上下文）与岗位作答，**可多次**，问答**追加在结果之后**。落库为 `LearnerQuestion`（挂在该次提交下），打分服务新增 `answerQuestion` 契约（Mock 确定性），并同样进 AI 审计日志（phase=answerQuestion）。
- 字数：笔记 **100–2000 字**（原 5000 收紧）；追问回答**每条 ≤ 1000 字**；初评分提交后在答追问页显著显示。
- 长文收纳：同伴笔记与答题记录里的长文**默认全部展开**，超过约 3 行才出现「收起 / 展开全文」开关（按钮统一右下角），方便直接通读、需要时再折叠。
- **流式输出**：追问回答**逐字流式**返回（Node 路由 `/api/learn/ask` + DeepSeek SSE；Mock 切片模拟），客户端实时显示；整段生成完才落库。流式调用同样写 AI 审计日志。
- 同伴可见（「同事怎么写的」）展示对方的**完整记录**：笔记 + 每条追问与回答（不含对方的 AI 反馈，那是个人评价不公开）。
- **思考过程**：追问回答把 DeepSeek 的 `reasoning_content` 也流式带回并展示——生成中实时显示，完成后收进「思考过程」可折叠区（落库 `LearnerQuestion.reasoning`，刷新仍在）。
- **提交笔记 / 答追问也改为流式**（Node 路由 `/api/learn/submit`、`/api/learn/finalize`）：等待时实时显示 AI 思考过程；但这两步最终是**结构化 JSON**（分数/追问/反馈），所以**只流式展示 reasoning，content(JSON) 在后台累计、解析、落库，绝不外泄**；完成后 `router.refresh()` 进入下一步。Mock 无 reasoning，直接完成。
- **初评 / 终评思考过程可回看**：提交 / 终评流式时累计的 reasoning **落库**（`Scoring.initialReasoning` / `Scoring.finalReasoning`），刷新后仍可在**分数旁的小按钮**（小弹窗 Dialog）里查看完整推理——答追问页（初评分旁）、结果页（终评分旁）、及答题记录每条（初评 / 终评徽标旁）。Mock / 历史无 reasoning 时不显示按钮。
- **Markdown 渲染**：AI 回答 / 反馈、以及笔记等内容按 Markdown 渲染（`.md` 克制排版）。长文默认全部展开，超过约 3 行才出现「收起 / 展开全文」（纯文本按行截断、Markdown 按高度截断，按钮在右下角且 `cursor:pointer`）。
- **成长轨迹画像全文**：成长轨迹 / 我的资料里的「画像全文」原为等宽 `<pre>` 折叠，改为 **Markdown 渲染 + 默认展开**（画像本就是带 `#`/`##`/列表 的 Markdown，结构化阅读更清楚）。
- **成长轨迹入口多处可达**：除顶栏「成长」外，闯关首页头部常驻「画像更新 N 次 · 看轨迹」入口（有画像才显示）、排行榜「我的名次」下「看我的成长轨迹」、以及我的资料 / 单词结果页 / 章节反思小结处均可进入，方便随时回看画像演进。
